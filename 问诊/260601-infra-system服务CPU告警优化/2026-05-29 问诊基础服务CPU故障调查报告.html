2026-05-29 问诊基础服务CPU故障调查报告

本报告基于线程栈、容器资源现象、旧实例连续 dump、以及 Arthas 在线观测结果整理。 调查对象为 saas-inquiry-user-server 旧实例与新实例在同等流量下的 CPU 差异。

服务 saas-inquiry-user-server
JDK 21.0.5
容器 CPU request 4 / limit 6
主要热点 JSqlParser Cache

一、最终结论

直接原因 CPU 突刺发生在 MyBatis-Plus 数据权限 SQL 解析链路： DataPermissionInterceptor -> JsqlParserGlobal.parse -> JsqlParseCache.getStatement。

关键差异 新旧实例请求量和 SQL 长度接近，但旧实例单次 parse() 平均耗时更高， 主要成本集中在 JdkSerialCaffeineJsqlParseCache.getStatement()。

综合判断：旧实例长期运行并经历大量医保目录更新后，JSqlParser 解析缓存和堆内对象状态变重， 导致 Caffeine + JDK 序列化缓存查询/恢复解析结果的成本升高。同等流量下，旧实例 CPU 更容易突刺。

二、故障现象

  • 旧实例一旦重新承接流量，CPU 明显高于新实例。
  • top -H -p <pid> 观察到高 CPU 线程为 mybatis-plus-jsqlParser-*。
  • 新实例承接当前线上流量后，CPU 没有同等程度突刺。
  • 旧实例停流量后，连续线程栈显示 JSqlParser 线程池逐步回到空闲状态。

这说明当前 JSqlParser 缓存使用 Caffeine，并结合 JDK 序列化方式保存或恢复解析结果。 命中缓存时仍可能产生反序列化、深拷贝、对象重建等 CPU 成本。

四、原因分析

  1. 昨晚更新 30-40 万医保目录后，旧实例长时间运行过程中可能积累了大量不同 SQL 解析缓存和 AST 对象。
  2. 早上用户并发上来后，大量 Dubbo 请求进入数据权限拦截器，每次查询前都会调用 JsqlParserGlobal.parse()。
  3. 新旧实例 parse 调用量接近，说明不是明显请求量不均。
  4. 新旧实例 SQL 长度接近，说明不是 SQL 明显变长。
  5. 旧实例 parse 平均耗时更高，且主要耗时集中在 JsqlParseCache.getStatement()。
  6. 旧实例缓存规模、缓存对象状态或堆内对象分布较新实例更重，导致 Caffeine + JDK 序列化缓存查询/恢复成本升高。

五、为什么新实例正常

新实例是冷启动状态，JSqlParser 缓存较小，堆内对象状态更干净。 即使承接相近流量，JsqlParserGlobal.parse() 单次平均耗时更低，因此 CPU 没有明显突刺。

重启旧实例能缓解问题，本质上是清空 JVM 内 JSqlParser 缓存和相关对象状态。 这能止血，但不是根治。

七、影响范围

  • 旧实例 CPU 使用率升高，可能触发容器 CPU limit 下的调度竞争。
  • Dubbo 请求处理线程在 SQL 执行前消耗更多 CPU。
  • 数据权限拦截涉及的高频查询会被放大影响。
  • 如果缓存继续膨胀或抖动，可能出现周期性突刺。

八、处置建议

短期止血

  1. 重启旧实例，清空 JVM 内 JSqlParser 缓存和相关对象状态。
  2. 临时摘除 CPU 异常旧实例，优先由新实例承接流量。
  3. 持续观察 monitor JsqlParserGlobal parse -c 5 的 avg-rt。

根治方向

  1. 对不需要数据权限改写的查询，使用 @InterceptorIgnore(dataPermission = "true")，让拦截器在进入 JSqlParser 前直接返回。
  2. 当前优先关注链路： UserRoleMapper.getUserListByRoleIds、 PermissionServiceImpl.getUserListByRoleIds、 AdminUserApiImpl.getUserListByRoleCodes。
  3. 将 mapper default 方法改成明确 mapped statement，再挂拦截器忽略注解，避免注解挂不到最终 mappedStatement。
  4. 评估 JSqlParser 缓存实现和容量，避免长期运行后缓存膨胀或频繁淘汰。
  5. 对高频权限类查询增加业务缓存，减少进入 MyBatis-Plus 数据权限解析链路的次数。

十、最终判断

本次故障的根因不是单条 SQL 慢、数据库慢或请求量明显不均，而是旧实例高频进入 MyBatis-Plus 数据权限 SQL 解析链路时，JdkSerialCaffeineJsqlParseCache.getStatement() 在旧实例上的平均成本显著高于新实例，导致 CPU 突刺。

最有效的长期治理是减少不必要的 DataPermissionInterceptor -> JsqlParserGlobal.parse 调用，优先对无需数据权限的 mapper 查询做 @InterceptorIgnore(dataPermission = "true")。
