问诊基础服务 CPU 优化两阶段实施方案

本文将《问诊基础服务 CPU 优化改造方案报告》和 JSqlParser 缓存调优建议合并成可执行的两阶段方案。 主线原则是：第一阶段严格按原报告做单点 Mapper 数据权限短路；第二阶段只在第一阶段后仍存在明确 JSqlParser 缓存热点时，再做全局缓存可配置化与调优。

1. 结论与优先级

最终结论 优先执行原优化报告中的第一阶段方案：只对 UserRoleMapper.getUserListByRoleIds 做精确的 @InterceptorIgnore(dataPermission = "true") 短路，并通过 XML MappedStatement 保证注解命中。 JSqlParser 缓存调优必须作为第二阶段，在确认第一阶段后仍有 JsqlParseCache.getStatement() 明确热点时再实施。
阶段 方案 目标 影响面 优先级
第一阶段 UserRoleMapper.getUserListByRoleIds 精确跳过 DataPermission 减少高频无效进入 DataPermissionInterceptor -> JsqlParserGlobal.parse 的 SQL 单个 Mapper 方法，风险可边界化验证 必须优先
第二阶段 JSqlParser cache 可配置化、指标化、灰度调优 降低仍然进入 parse 链路后的缓存命中/反序列化/重复解析成本 全局 SQL 解析缓存，影响所有相关插件 按观测结果触发
为什么不先调缓存？ 缓存调优不能减少 JsqlParserGlobal.parse() 的入口次数；而故障报告的热点链路来自数据权限 SQL 解析。 对于本来不需要数据权限解析的 system_user_role 查询，正确做法是让它不要进入该解析链路，而不是优先扩大全局缓存。

2. 背景与故障链路

2.1 已知故障链路

DubboServerHandler
  -> PermissionServiceImpl.getUserListByRoleIds
  -> UserRoleMapper.getUserListByRoleIds
  -> DataPermissionInterceptor.beforeQuery
  -> JsqlParserGlobal.parse
  -> JsqlParseCache.getStatement
  -> JdkSerialCaffeineJsqlParseCache.getStatement

2.2 关键判断

  • 旧实例与新实例请求量、SQL 长度接近，但旧实例 parse() 平均耗时更高。
  • 高 CPU 线程集中在 mybatis-plus-jsqlParser-* 一类线程与 JSqlParser 解析链路。
  • 当前工程依赖 MyBatis-Plus 3.5.12，yudao 默认配置中 JSqlParser cache 可能为 JdkSerialCaffeineJsqlParseCache(maximumSize=1024, expireAfterWrite=5s)。
  • JdkSerialCaffeineJsqlParseCache 命中缓存时仍需要从 byte[] 反序列化 AST，命中并不是零成本。
  • 当前可见的数据权限配置未体现 system_user_role 的部门/本人数据范围规则，因此目标查询很可能属于高频无效解析。
本方案的优化方向 第一阶段减少无效 parse 入口；第二阶段优化残余 parse 的缓存成本。两者不是替代关系，而是先后关系。

3. 总体范围与禁区

3.1 本次纳入范围

范围 说明

3.2 硬性禁区

以下做法禁止作为本次优化手段
  • 禁止无条件关闭全局数据权限插件；若采用 6.2 条件化关闭方案，必须先确认生产运行时不存在有效 DataPermissionRule 表配置，并配置启动保护。
  • 禁止添加 @TenantIgnore。
  • 禁止添加 @InterceptorIgnore(tenantLine = "true")。
  • 禁止把 system_user_role、system_tenant_user_relation 加入租户插件 ignore-tables。
  • 禁止修改 AdminUserMapper.getTenantAvailableUserList 作为第一阶段内容。
  • 禁止 blanket-ignore 系统用户、员工列表、角色用户等宽范围 SQL。
  • 禁止通过 TenantUtils.executeIgnore 或类似方式修复空结果。

3.3 语义必须保持

  • 租户隔离：仍由 TenantLineInnerInterceptor 追加租户条件。
  • 逻辑删除：从 BaseMapper 切换到 XML 后，必须显式保留 deleted = false 或等价条件。
  • 空集合保护：Service 层继续提前返回空；XML 层也必须 fail-closed。
  • 数据范围：getUserListByRoleIds 只返回用户角色关系，不代表当前登录人可见员工列表；部门/本人范围仍走原有专用链路。

4. 第一阶段：UserRoleMapper 精确短路

阶段目标 将 UserRoleMapper.getUserListByRoleIds 从 Java default helper 改为明确的 MyBatis MappedStatement， 使 @InterceptorIgnore(dataPermission = "true") 能准确命中该 SQL，跳过 DataPermission SQL 解析，同时继续保留租户插件和逻辑删除约束。

4.1 当前问题

当前方法如果是 Java default helper，并在内部委托 BaseMapper.selectList，则注解加在 default 方法上不一定能作用到最终执行的 MappedStatement。 实际执行的 mappedStatementId 可能仍是 BaseMapper 相关语句，导致 @InterceptorIgnore 不命中，CPU 优化无效。

4.2 Mapper 改造

将 UserRoleMapper.getUserListByRoleIds 改为抽象方法，并只添加数据权限忽略：

import com.baomidou.mybatisplus.annotation.InterceptorIgnore;
import org.apache.ibatis.annotations.Param;

@InterceptorIgnore(dataPermission = "true")
List<UserRoleDO> getUserListByRoleIds(@Param("roleIds") Set<Long> roleIds);
注意 这里不能保留 default 方法体；否则可能继续委托到 BaseMapper.selectList，导致注解没有命中真实 SQL。

4.3 XML MappedStatement

新增或修改 UserRoleMapper.xml，确保 namespace、select id、参数名完全匹配。

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="cn.iocoder.yudao.module.system.dal.mysql.permission.UserRoleMapper">

    <select id="getUserListByRoleIds"
            resultType="cn.iocoder.yudao.module.system.dal.dataobject.permission.UserRoleDO">
        select *
        from system_user_role
        <where>
            deleted = false
            <choose>
                <when test="roleIds != null and roleIds.size() > 0">
                    and role_id in
                    <foreach collection="roleIds" item="roleId" open="(" separator="," close=")">
                        #{roleId}
                    </foreach>
                </when>
                <otherwise>
                    and 1 = 0
                </otherwise>
            </choose>
        </where>
    </select>

</mapper>

XML 要点

  • namespace 必须是 cn.iocoder.yudao.module.system.dal.mysql.permission.UserRoleMapper。
  • select id 必须是 getUserListByRoleIds。
  • Mapper 参数必须使用 @Param("roleIds")，XML 中 collection="roleIds"。
  • 必须显式包含 deleted = false 或数据库实际等价条件。
  • 空集合必须 and 1 = 0，不能退化成全租户或全表查询。
  • 不要手写 tenant_id 条件，租户条件必须继续由 TenantLineInnerInterceptor 追加。

4.4 Service 层保护

保留现有空集合提前返回逻辑：

public List<UserRoleVO> getUserListByRoleIds(Set<Long> roleIds) {
    if (CollUtil.isEmpty(roleIds)) {
        return new ArrayList<>();
    }
    List<UserRoleDO> userListByRoleIds = userRoleMapper.getUserListByRoleIds(roleIds);
    return UserRoleConvert.INSTANCE.convertVo(userListByRoleIds);
}

5.5 第一阶段验收标准

  • 最终执行的 mappedStatementId 是 cn.iocoder.yudao.module.system.dal.mysql.permission.UserRoleMapper.getUserListByRoleIds。
  • InterceptorIgnoreHelper.willIgnoreDataPermission(mappedStatementId) 返回 true。
  • 目标 mappedStatement 即使出现在 DataPermissionInterceptor.beforeQuery 采样中，也必须在命中 ignore 后快速返回，不再从 DataPermission 链路调用 JsqlParserGlobal.parse()。

  • 空 roleIds 返回空，不产生全表/全租户查询。
  • system_user_role 查询执行计划合理，慢 SQL、扫描行数、DB span 无异常放大。

  • CPU、接口 P95/P99 较旧实例改善，且无新增功能异常。

6. 阶段间观察与决策门槛

第一阶段上线后，不应立即进入第二阶段。需要先收集残余热点，判断是否仍然有必要对全局 JSqlParser cache 动手。

6.1 需要收集的数据

数据 用途
仍从 DataPermissionInterceptor.beforeQuery 下钻到 JsqlParserGlobal.parse 的 mappedStatementId TopN 识别是否还有其他高频无效数据权限解析 SQL；仅进入 beforeQuery 但命中 ignore 后快速返回的不计入。
生产运行时 DataPermissionRuleFactory 中实际生效的规则与 tableNames 确认当前 DataPermission 是否存在“有拦截器、无有效表规则”的全局 no-op parse 情况。
JsqlParserGlobal.parse total / avg-rt / max-rt 判断 parse 总量和平均耗时是否已经回到可接受范围。
JsqlParseCache.getStatement 慢调用 判断残余热点是否仍在缓存查询/反序列化。
CPU 火焰图 / top thread 确认热点是否仍为 mybatis-plus-jsqlParser-* 或 JSqlParser 相关栈。
Heap / Old GC / Full GC 判断是否存在缓存对象或 AST 反序列化造成的堆压力。
阶段间关键判断 如果运行时规则显示 DataPermission 没有任何有效 tableNames，说明当前拦截器可能对大量 SELECT 只做 SQL parse、但不追加任何权限条件。 这种情况不应直接扩大缓存作为唯一治理手段，应单独评估“条件化注册/执行 DataPermissionInterceptor”或继续按 mappedStatementId 做安全短路。

6.2 补充方案：DataPermissionInterceptor 条件化关闭

方案定位 这是第一阶段后的阶段间治理分支，不是替代第一阶段的默认动作，也不是无条件关闭数据权限。 仅当生产运行时确认 DataPermissionRule 没有任何有效 tableNames 时，才允许让 DataPermissionInterceptor 不注册或被显式移除；一旦存在有效规则，必须注册拦截器或启动失败，避免静默越权。

6.2.1 现状依据

  • YudaoDataPermissionAutoConfiguration 会创建 DataPermissionRuleHandler，再把 MyBatis-Plus 的 DataPermissionInterceptor 加入 MybatisPlusInterceptor 第 0 位。
  • MyBatis-Plus 3.5.12 中，非忽略的 SELECT 会先进入 parserSingle()，UPDATE/DELETE 会先进入 parserMulti()；即使后续规则返回 null，也已经触发 JSqlParser 解析。
  • 当前工作区 infra-user 的 DataPermissionConfiguration 中 rule.addDeptColumn(...)、rule.addUserColumn(...) 均为注释状态，全仓也未发现启用中的同类配置。
  • @DataPermission(enable = false)、DataPermissionUtils.executeIgnore(...)、@InterceptorIgnore(dataPermission = "true") 主要用于关闭或短路数据权限；当没有有效表规则时，它们不会改变结果集，但仍可能保留历史语义。
  • 最终判断必须以生产运行时 Bean 为准：启动时应打印并校验所有 DataPermissionRule#getTableNames()，不能只依赖源码 grep。

6.2.2 推荐改动方案

步骤 改动 要求
1. 保留依赖 继续保留 yudao-spring-boot-starter-biz-data-permission 相关传递依赖。 不能直接删除依赖；当前代码仍 import @DataPermission、DataPermissionUtils、DeptDataPermissionRuleCustomizer 等类型。
2. 排除原自动注册 在目标服务中排除 cn.iocoder.yudao.framework.datapermission.config.YudaoDataPermissionAutoConfiguration。 只排除“注册 DataPermissionInterceptor 的自动配置”；不要排除租户插件、分页插件、MyBatis 基础配置。
3. 本地补回兼容 Bean 新增本地配置类，继续提供 DataPermissionRuleFactory、DataPermissionRuleHandler、DataPermissionAnnotationAdvisor。 保证已有注解、工具类和潜在注入点不因自动配置排除而缺 Bean。
4. 条件化注册拦截器 新增配置项，例如 inquiry.mybatis-plus.data-permission.interceptor-mode=auto|on|off。 on：始终注册；auto：存在有效 tableNames 才注册；off：不注册，但发现有效规则时启动失败。
5. 启动保护 启动日志输出 mode、规则类、tableNames、最终是否注册 DataPermissionInterceptor。 如果 off 或 auto 决定不注册，但存在非空 tableNames，必须 fail-closed，不能仅 WARN 后继续启动。

6.2.3 不推荐的做法

  • 不推荐直接删除 starter 或依赖树中的数据权限 jar，容易导致编译期缺类。
  • 不推荐在公共框架层全局硬编码关闭，避免影响后续启用部门数据权限的服务。
  • 不推荐只在启动后粗暴 removeIf(DataPermissionInterceptor) 且没有规则校验；这会留下启动期 SQL、初始化顺序和静默越权风险。
  • 不得同时关闭 TenantLineInnerInterceptor、分页插件或逻辑删除相关能力。

6.2.4 影响范围

维度 影响 风险等级 验证要求
编译依赖 保留 starter 时无预期编译影响；直接删除依赖会影响 @DataPermission、DataPermissionUtils、DeptDataPermissionRuleCustomizer 等 import。 中 执行 mvn -pl infra-user/infra-user-server -am test 或至少 compile。
当前 infra-user-server 查询结果 在运行时规则 tableNames 为空的前提下，SQL 结果集理论上不变；原拦截器只产生解析成本，不追加权限条件。 低 对比员工、部门、角色、用户角色关系接口结果；重点校验 A/B 租户负样本。
数据权限安全 如果未来或其它服务启用了 addDeptColumn / addUserColumn / 自定义 DataPermissionRule，关闭拦截器会导致 DB 层部门/本人范围失效。 高 启动时发现任何有效 tableNames 必须注册拦截器或启动失败。
注解语义 @DataPermission(enable = false)、DataPermissionUtils.executeIgnore(...)、@InterceptorIgnore(dataPermission = "true") 在不注册拦截器时变成冗余语义。 中 保留注解并在文档/日志中说明“当前 no-op”，不要大范围清理历史注解。
租户隔离 不应受影响；租户隔离仍由 TenantLineInnerInterceptor 负责。 中 检查 MyBatis interceptor 链仍包含租户插件；采样最终 SQL 仍有 tenant_id 条件。
逻辑删除 不应受影响；逻辑删除来自 MyBatis-Plus 表元数据或 XML 显式条件。 低 验证 deleted=true 数据不返回。
性能 可减少所有非必要 DataPermission 链路的 JSqlParser parse/cache get/put；但租户、分页等插件仍可能触发 JSqlParser，parse 不会归零。 正向 对比 DataPermissionInterceptor -> JsqlParserGlobal.parse 调用量、CPU、接口 P95/P99。
其它模块 saas-crm 存在少量 @DataPermission(enable = false)；saas-inquiry-kernel、saas-inquiry-signature、saas-transmitter 当前搜索未见直接使用。 低到中 不要把该配置默认下沉到公共依赖；每个启动服务单独检查运行时规则。

6.2.5 验证与回滚

6.2.8 直接禁用 DataPermissionInterceptor 的影响范围评估

方案 改动范围 收益 主要风险 结论
继续逐个热点短路 每个热点 Mapper 方法单独改 Java + XML + 测试。 风险最可控，只影响选中的 mappedStatement。 候选数量大，治理成本高；只能减少被点杀 SQL 的 DataPermission parse。 适合第一阶段和后续 TopN 精准治理。
条件化禁用 DataPermissionInterceptor 每个启动服务一处配置/本地自动配置；无需改 704 个 Mapper 方法。 一次性减少该服务所有无效 DataPermission parse/cache get/put。 必须证明运行时没有有效 DataPermissionRule#tableNames；未来启用规则时必须 fail-closed。 推荐作为阶段间治理分支，但必须有启动保护。
无条件禁用 DataPermissionInterceptor 全局影响所有 Mapper、XML 和 BaseMapper SQL。 性能收益最大、实施最简单。 一旦存在部门/本人数据权限规则，会导致所有 DB 层数据范围静默失效；注解语义也会误导维护者。 不推荐。
直接禁用的边界不是 291 或 704 个方法

DataPermissionInterceptor 是 MyBatis-Plus 全局 InnerInterceptor。关闭它后，影响面不是“类似 default helper”的候选数量， 而是该 JVM 内所有会经过 MyBatis-Plus 拦截器链的 SELECT、UPDATE、DELETE，包括 BaseMapper 自动 SQL、Mapper default helper、XML MappedStatement、以及依赖 jar 中被当前应用扫描到的 Mapper。

在当前工作区没有有效 DataPermissionRule 表配置的前提下，直接禁用预计主要改变性能，不改变结果集；但如果运行时加载了带有效表配置的 yudao/system 代码，或未来恢复 addDeptColumn、addUserColumn，则会变成跨部门/本人数据范围失效的安全问题。

运行时差异提醒 当前工作区的 infra-user 数据权限配置中 addDeptColumn/addUserColumn 为注释状态；但本地 Maven 仓库中的 yudao-module-system-2025.08-SNAPSHOT 源码曾发现存在启用版配置。 因此不能只看源码扫描结论，必须在目标服务启动时打印实际加载的 DataPermissionRule 列表和 tableNames，并以运行时结果决定是否注册 DataPermissionInterceptor。

6.3 进入第二阶段的建议门槛

满足以下任意组合，再进入第二阶段
  • 第一阶段后 JsqlParserGlobal.parse avg-rt 仍显著高于新实例基线。
  • Arthas trace 显示慢调用仍集中在 JsqlParseCache.getStatement() 或缓存反序列化。
  • DataPermission 无效解析已明显下降，但 JSqlParser 线程仍是 CPU Top 热点。
  • 剩余高频 mappedStatementId 无法通过第一阶段同类手段安全短路。

6.4 不进入第二阶段的情况

  • CPU 已恢复稳定，接口耗时满足目标。
  • parse total / avg-rt 已回到新实例基线。
  • 残余热点转移到业务逻辑、DB、RPC、GC 等非 JSqlParser 链路。
  • 无法建立 JSqlParser cache 指标或无法稳定灰度对比。

7. 第二阶段：JSqlParser 缓存可配置化

阶段定位 第二阶段不是替代第一阶段，也不是盲目扩大缓存。它的目标是：在残余热点仍集中于 JSqlParser cache 时，将原先硬编码且不可观测的缓存改为可配置、可观测、可回滚的全局缓存策略。

7.1 推荐落点

infra-user/infra-user-server/src/main/java/com/xyy/saas/inquiry/user/server/config/mybatis/JsqlParserCacheProperties.java
infra-user/infra-user-server/src/main/java/com/xyy/saas/inquiry/user/server/config/mybatis/JsqlParserCacheConfiguration.java

选择服务侧配置类而不是改依赖源码的原因：

  • 本次故障服务是 saas-inquiry-user-server。
  • 应用启动类已扫描 com.xyy.saas.inquiry。
  • 不污染 MyBatis-Plus 或 yudao 依赖，便于升级和回滚。

7.2 配置项设计

inquiry:
  mybatis-plus:
    jsql-parser-cache:
      mode: tuned       # legacy | tuned | off
      maximum-weight: 64MB
      expire-after-access: 10m
      record-stats: true
配置 含义 建议
mode=legacy 恢复 yudao 默认风格：JDK 序列化 + 1024 条 + 5 秒 write TTL。 回滚使用。
mode=tuned 使用调优配置：JDK 序列化 + 权重上限 + 热点访问续期 + 统计。 第二阶段首选。
mode=off 关闭 JSqlParser parse cache，每次重新解析。 仅压测/紧急对比，不建议长期生产。
maximum-weight=64MB 以 SQL key + AST 序列化 byte[] 近似大小控制缓存，而不是单纯按条数。 首发 64MB，后续按 heap/GC 与命中率评估是否到 128MB。
expire-after-access=10m 热点 SQL 访问后续期，减少 5 秒固定过期造成的重复 parse。 首发 10 分钟。
record-stats=true 开启 Caffeine stats，支持命中率、miss、eviction 观测。 灰度必须开启。

7.3 tuned 模式核心实现示意

Cache<String, byte[]> nativeCache = Caffeine.newBuilder()
        .maximumWeight(properties.getMaximumWeight().toBytes())
        .weigher((String key, byte[] value) -> {
            int keyWeight = key == null ? 0 : key.length() * Character.BYTES;
            int valueWeight = value == null ? 0 : value.length;
            return Math.max(1, keyWeight + valueWeight);
        })
        .expireAfterAccess(properties.getExpireAfterAccess())
        .recordStats()
        .build();

JsqlParserGlobal.setJsqlParseCache(new JdkSerialCaffeineJsqlParseCache(nativeCache));

实现原则

  • 首发仍使用 JdkSerialCaffeineJsqlParseCache，不同时引入 Fury/FST。
  • 不要缓存原始 Statement 对象；序列化/反序列化本质上提供 AST 深拷贝，避免插件修改污染缓存对象。
  • 配置类调用 JsqlParserGlobal.setJsqlParseCache(...) 后会覆盖 yudao 静态初始化的 legacy cache；必须确保在业务 SQL 执行前完成初始化。
  • 必须保留 nativeCache 引用，用于注册 Micrometer 指标；不能只创建 JdkSerialCaffeineJsqlParseCache 后丢失原生 Caffeine cache。
  • maximum-weight 只约束 SQL key + 序列化 byte[] 的近似权重，不等于 JVM 实际 heap 占用；灰度必须同时观察 allocation rate、Old Gen 和 GC pause。
  • 不要做动态刷新；缓存策略调整后通过重启灰度实例生效。
  • 启动日志必须打印最终生效的 cache mode、capacity/weight、TTL、stats 配置。
  • 保留 legacy 和 off 作为回滚/对比开关。

7.4 指标设计

指标 说明
jsqlparser.cache.estimated.size 当前缓存条目数。
jsqlparser.cache.hit.count 缓存命中次数。
jsqlparser.cache.miss.count 缓存未命中次数。
jsqlparser.cache.hit.rate 命中率。
jsqlparser.cache.eviction.count 缓存淘汰次数。
jsqlparser.cache.eviction.weight 淘汰权重，用于判断缓存容量是否过小或 key 过度发散。
指标注册要求 JdkSerialCaffeineJsqlParseCache 本身不直接暴露 Micrometer 指标。实现时应保留 Cache<String, byte[]> nativeCache，通过 CaffeineCacheMetrics.monitor(...) 或等价方式注册指标， 并在预发确认 hit/miss/eviction/estimatedSize 可真实查询。

7.5 A/B 对比矩阵

组别 配置 目的 是否推荐生产首发
A 基线 JDK + size 1024 + write TTL 5s 当前或 legacy 基线。 回滚基线
B Tuned-1 JDK + weight 64MB + access TTL 10m + stats 降低热点 SQL 反复过期和重复 parse。 是
C Off 关闭 JSqlParser cache 判断 JDK 反序列化是否比重新 parse 更贵。 否，仅压测/临时对比
D Fury FurySerialCaffeineJsqlParseCache + 同等 cap/TTL 验证是否能降低序列化/反序列化 CPU。 否，后续单独验证
关于 Fury / FST MyBatis-Plus 3.5.12 已提供 Fury/FST 相关缓存实现，但依赖是 optional。考虑 Java 21 兼容性、依赖引入和序列化行为变化，第二阶段首发不建议直接切换 Fury/FST。 如果 Tuned-1 后 getStatement() 仍是明确热点，再单独立项做预发 A/B。

8. 第二阶段验证与灰度

8.1 第二阶段预发验证

  1. 启动后确认日志输出最终 JSqlParser cache mode 与参数。
  2. 确认 Micrometer / 日志可以看到 cache hit/miss/eviction/estimatedSize，且这些指标来自当前生效的 native Caffeine cache。
  3. 回放或压测高频权限查询链路。
  4. 在同等流量与同等数据集下，对比 legacy 与 tuned 的 parse avg-rt、慢 parse 数量、CPU、heap、allocation rate、GC。
  5. 执行第一阶段全部功能回归，确认 SQL 语义无变化。

8.2 第二阶段生产灰度

Step 1：单实例 tuned

仅一个生产实例启用 mode=tuned，其它实例保持 legacy，便于横向对比。

Step 2：30 分钟启动稳定性观察

确认无 SQL parse 异常、序列化异常、启动异常、内存快速增长。

Step 3：60 分钟正常流量观察

观察 parse avg-rt、hit rate、eviction、CPU、P95/P99、Old GC。

Step 4：高峰验证后扩容

覆盖一个业务高峰后，再按 30% / 50% / 全量推进。

8.3 第二阶段验收标准

维度 目标
parse avg-rt 较 legacy 明显下降；目标接近新实例基线，或至少下降 30%。
慢 parse #cost > 1ms 的数量明显下降。
CPU JSqlParser 相关线程不再持续作为 Top 热点。
Cache hit rate 稳定，eviction 不持续高速增长，native cache 指标与启动日志中的 mode/参数一致。
Heap / GC Old Gen、Full GC、GC pause、allocation rate 无明显恶化。
业务语义 租户隔离、逻辑删除、数据权限回归全部通过。

8.4 第二阶段调参规则

  • 如果 hit rate 低、eviction 高、heap 稳定，可从 64MB / 10m 小步调整到 128MB / 30m。
  • 如果 hit rate 高但 getStatement() 仍慢，不要继续盲目加容量，应评估序列化实现或继续减少 parse 入口。
  • 如果 heap / GC 变差，立即降回 64MB 或 legacy。
  • 如果 SQL key 基数过高，应优先治理 SQL 生成方式和 IN 列表规模，而不是无限加缓存。

9. 回滚策略

9.1 第一阶段回滚

触发条件 跨租户数据返回、deleted=true 关系返回、部门/本人数据范围扩大、BindingException、Invalid bound statement、XML 加载异常、DB 慢 SQL 或扫描行数异常、CPU 无改善且出现新瓶颈。

回滚要求：

  • 必须整体回滚 Java Mapper 与 XML，不能只回滚 Java 或只回滚 XML。
  • 不得通过 @TenantIgnore、tenantLine = "true"、租户 ignore-tables 或全局关闭数据权限来止血。
  • 回滚后重启或替换实例，清理本地 JSqlParser parse cache 与类加载状态。

9.2 第二阶段回滚

通过配置切回 legacy，并重启实例：

inquiry:
  mybatis-plus:
    jsql-parser-cache:
      mode: legacy

紧急对比或缓存疑似异常时，可临时关闭缓存：

inquiry:
  mybatis-plus:
    jsql-parser-cache:
      mode: off
注意 mode=off 可能导致每次 SQL 都重新 parse，CPU 反而上升，只能作为短期压测或故障定位手段，不建议长期生产使用。
