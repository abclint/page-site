问诊基础服务 CPU 优化改造方案报告

本报告基于 2026-05-29 问诊基础服务 CPU 故障调查报告 与当前代码整理， 目标是在不破坏租户隔离、逻辑删除和业务数据范围的前提下，减少高频无效进入 MyBatis-Plus 数据权限 SQL 解析链路的次数。

服务 saas-inquiry-user-server
主要热点 DataPermission + JSqlParser
首批改造对象 UserRoleMapper.getUserListByRoleIds
报告日期 2026-05-30

一、执行摘要

故障根因 线上 CPU 突刺发生在 DataPermissionInterceptor -> JsqlParserGlobal.parse -> JsqlParseCache.getStatement 链路。 旧实例与新实例请求量、SQL 长度接近，但旧实例单次解析缓存访问成本更高。

改造方向 先减少无必要的数据权限 SQL 解析，再评估 JSqlParser 缓存和业务缓存。 第一阶段只针对 system_user_role 用户-角色关系查询做局部短路。

首批目标 SQL 1 条
改造风险等级 中高
必须保留 租户隔离
必须验证 SQL 语义

本方案不是通用的“关闭数据权限”方案，而是只绕过 UserRoleMapper.getUserListByRoleIds 这一条角色关系查询的 DataPermissionInterceptor 解析。租户插件 tenantLine、逻辑删除条件和下游员工可见性过滤必须保持原有语义。

二、现状与代码证据

1. 故障报告中的热点链路

2. 当前首批热点方法是 default helper

当前 getUserListByRoleIds 不是明确的 MyBatis mapped statement，而是 Java default 方法，内部委托 BaseMapper 的 selectList。 

// infra-system/infra-user/infra-user-server/src/main/java/cn/iocoder/yudao/module/system/dal/mysql/permission/UserRoleMapper.java

default List<UserRoleDO> getUserListByRoleIds(Set<Long> roleIds) {
    return selectList(UserRoleDO::getRoleId, roleIds);
}

风险点：如果只在该 default 方法上加 @InterceptorIgnore(dataPermission = "true")，运行时真正执行的 MappedStatement 仍可能是 BaseMapper 注入的 selectList，注解无法稳定命中热点 SQL。

3. 调用链

代码位置 职责 改造判断
infra-system/.../api/user/AdminUserApiImpl.java#getUserListByRoleCodes Dubbo/API 层入口，委托 AdminUserServiceImpl.getUserListByRoleCodes。 不是 SQL mapped statement，不应在这里加 @InterceptorIgnore。
infra-system/.../api/user/AdminUserApiImpl.java#getCrmUserRoleIdListByRoleIds 通过 role code 查询 role，再通过 PermissionServiceImpl.getUserListByRoleIds 获取 userId。 调用场景需确认是否只需要当前租户角色成员。
infra-system/.../service/user/AdminUserServiceImpl.java#getUserListByRoleCodes 普通按角色查询可用用户：roleCodes -> roleIds -> userRole -> tenant available users。 可享受首批 mapper 短路收益，但不能替代最终用户状态/租户员工关系过滤。
infra-system/.../service/user/AdminUserServiceImpl.java#getUserListByRoleCodes_Permission 带当前登录人部门/本人数据范围过滤。 不能简化为“按角色查用户都忽略数据权限”。
infra-system/.../service/permission/PermissionServiceImpl.java#getUserListByRoleIds 调用 UserRoleMapper.getUserListByRoleIds 并转换为 UserRoleVO。 保留空集合直接返回空列表。
infra-system/.../dal/mysql/permission/UserRoleMapper.java#getUserListByRoleIds 查询 system_user_role 用户-角色关系。 首批唯一建议落点。

三、改造目标与非目标

目标

  • 减少 UserRoleMapper.getUserListByRoleIds 进入 DataPermissionInterceptor 后的 JSqlParser 解析成本。
  • 保留 system_user_role.deleted = false 逻辑删除语义。
  • 保留 tenantLine 租户隔离语义。
  • 让 @InterceptorIgnore(dataPermission = "true") 稳定挂到最终 MappedStatement。
  • 通过灰度和 Arthas/JFR 观测确认 CPU 热点下降。

非目标 / 禁止项

  • 不是关闭全局数据权限。
  • 不是忽略租户插件，不允许添加 @TenantIgnore。
  • 不允许添加 @InterceptorIgnore(tenantLine = "true")；这里的 true 表示“忽略 tenantLine”。
  • 不改 AdminUserMapper.getTenantAvailableUserList。
  • 不对 system_users 或员工列表类 SQL 做 blanket ignore。

四、第一阶段推荐改造方案

1. 改造 UserRoleMapper：default 方法改成明确 mapped statement

建议保留方法名 getUserListByRoleIds，删除 default 方法体，改成 MyBatis mapper 抽象方法，并只添加 @InterceptorIgnore(dataPermission = "true")。 

import com.baomidou.mybatisplus.annotation.InterceptorIgnore;

@InterceptorIgnore(dataPermission = "true")
List<UserRoleDO> getUserListByRoleIds(@Param("roleIds") Set<Long> roleIds);

2. 改造 UserRoleMapper.xml：新增同名 select

XML 的 namespace 必须保持： cn.iocoder.yudao.module.system.dal.mysql.permission.UserRoleMapper。 select id 必须与 Java 方法名完全一致。 

<select id="getUserListByRoleIds"
        resultType="cn.iocoder.yudao.module.system.dal.dataobject.permission.UserRoleDO">
  select *
  from system_user_role
  <where>
    deleted = false
    <choose>
      <when test="roleIds != null and roleIds.size() > 0">
        and role_id in
        <foreach collection="roleIds" open="(" item="id" separator="," close=")">
          #{id}
        </foreach>
      </when>
      <otherwise>
        and 1 = 0
      </otherwise>
    </choose>
  </where>
</select>

注意 上述 SQL 显式写 deleted = false，租户条件不在 XML 中手写，继续交由 TenantLineInnerInterceptor 追加。 如果线上发现 tenantLine 未对该 SQL 追加租户条件，则必须暂停发布并先修正租户插件配置，不能通过 @TenantIgnore 规避。

3. PermissionServiceImpl 保持空集合保护

当前 PermissionServiceImpl.getUserListByRoleIds 已有空集合直接返回空列表逻辑，应保留。 

public List<UserRoleVO> getUserListByRoleIds(Set<Long> roleIds) {
    if (CollUtil.isEmpty(roleIds)) {
        return new ArrayList<>();
    }
    List<UserRoleDO> userListByRoleIds = userRoleMapper.getUserListByRoleIds(roleIds);
    return UserRoleConvert.INSTANCE.convertVo(userListByRoleIds);
}

4. 为什么不用“新方法名”作为首选

也可以新增 selectListByRoleIdsIgnoreDataPermission 并修改 service 调用，但首选保持 getUserListByRoleIds 的原因是：

  • 调用方无需改动，变更面更小。
  • Arthas 观测中的 mappedStatementId 更容易与故障报告中的热点链路对应。
  • 只需确保 default 方法体删除，XML id 与 Java 方法一一匹配。

五、改造后链路变化

改造前

PermissionServiceImpl.getUserListByRoleIds
  -> UserRoleMapper default getUserListByRoleIds
  -> BaseMapper selectList
  -> DataPermissionInterceptor.beforeQuery
  -> JsqlParserGlobal.parse
  -> JsqlParseCache.getStatement

改造后

PermissionServiceImpl.getUserListByRoleIds
  -> UserRoleMapper.getUserListByRoleIds MappedStatement
  -> @InterceptorIgnore(dataPermission = "true") 命中
  -> 跳过 DataPermissionInterceptor SQL 解析
  -> tenantLine 仍保持租户隔离

性能边界：本方案绕过的是该 MappedStatement 的数据权限解析，不承诺完全不走 JSqlParser。租户插件仍可能解析 SQL， 因此验收目标是 DataPermissionInterceptor 相关解析栈显著下降，而不是 JSqlParser CPU 归零。

六、风险评估与规避措施

风险类别 等级 风险说明 规避措施 验证方式
MappedStatement 未命中 高 如果仍保留 default 方法，或 XML id/namespace/参数名不一致，注解可能作用不到真正执行的 SQL，CPU 不下降，甚至出现 BindingException。 删除 default 方法体；Java 方法改抽象；XML id="getUserListByRoleIds"；参数统一 @Param("roleIds") 与 foreach collection="roleIds"。 启动时确认 MyBatis 存在该 statement；日志/Arthas 观察 mappedStatementId 是 UserRoleMapper.getUserListByRoleIds 而非 BaseMapper selectList。
租户隔离绕过 高 误加 @TenantIgnore、tenantLine = "true" 或把 system_user_role 加入 ignore-tables，会导致跨租户角色关系混入。 只允许 @InterceptorIgnore(dataPermission = "true")；禁止改租户配置；禁止 TenantUtils.executeIgnore 包裹该查询。 SQL/BoundSql 抽样确认 system_user_role 最终 SQL 含当前租户条件；A/B 租户同 roleId/roleCode 负样本互不可见。
逻辑删除失效 高 从 BaseMapper 切到 XML 后，手写 XML 不会自动继承原 LambdaQueryWrapper 的逻辑删除条件，可能返回已删除角色关系。 XML 固定写 deleted = false 或 deleted = 0，作为不可删除条件。 构造 deleted=true 的 system_user_role 数据，调用后不得返回；SQL 日志能看到 deleted 条件。
空集合退化为全量查询 高 如果 XML 只在 roleIds 非空时拼接 role_id in，空集合可能退化为当前租户全部角色关系。 Service 层继续空集合返回空；XML 层 otherwise and 1 = 0 fail-closed 双保险。 Mapper 级测试传 null、空 Set，必须返回空或受控异常，不能扫描全量。
数据范围误解 高 getUserListByRoleIds 返回的是用户-角色关系中间结果，不是当前登录人最终可见员工集合。若调用方直接返回，可能绕过部门/本人数据范围。 报告和代码注释明确该方法只返回当前租户角色成员关系；需要部门/本人过滤的场景必须走 getUserListByRoleCodes_Permission 或显式过滤。 对普通链路和 _Permission 链路分别做回归，验证部门受限账号看不到越权部门用户。
下游员工列表被误改 高 AdminUserMapper.getTenantAvailableUserList 查询 system_users 与 system_tenant_user_relation，员工租户隔离依赖 relation 表租户条件。若给该方法 blanket ignore，可能跨租户混入员工。 第一阶段不改 AdminUserMapper 与 getTenantAvailableUserList；未来如需优化必须另立方案。 静态扫描确认该 mapper 未新增 @TenantIgnore 或 tenantLine=true；最终 SQL 对 relation 表仍含租户条件。
租户上下文缺失 高 Dubbo、异步、BPM 或定时任务如果没有正确传递 TenantContextHolder，租户插件可能无法追加正确租户条件。 无租户上下文入口必须显式指定租户或 fail-closed，不能用 @TenantIgnore 修复空结果。 覆盖 HTTP、Dubbo、BPM/异步入口，日志确认 tenantId 非空且返回行租户一致。
性能预期过度 中 改造只绕过数据权限解析，tenantLine 仍可能解析 SQL；roleIds 数量不同会形成不同 SQL 形态和缓存 key。 目标表述为降低 DataPermission 解析成本；统计 roleIds size 分布，必要时限制单次数量或分批。 JFR/async-profiler 对比 DataPermission 栈下降；同时观察 tenantLine 残余解析、GC 分配和接口 P99。
数据库负载转移 中 应用 CPU 下降后，若 SQL 条件或索引不足，压力可能转移到数据库。 确认 SQL 包含 role_id in、deleted=false、租户条件；检查 system_user_role 索引。 预发执行 explain；线上观察慢 SQL、rows examined、DB span。
灰度混部误判 中 新旧实例并存时，旧实例仍走旧解析路径；Redis 缓存也可能导致验证请求不触发目标 mapper。 按实例版本、commit、pod、mappedStatementId 维度观测；验证时固定路由或构造首次查询。 灰度期间分别抽样新旧实例 SQL 和 CPU 栈；全量替换后再下最终结论。
回滚不完整 高 只回滚 Java 或只回滚 XML 会导致接口和 mapper 不匹配；用扩大 ignore 止血会带来越权。 只允许整体回退上一版本制品或关闭该精确 mapper 路径；禁止全局关闭数据权限/租户插件。 回滚后确认版本一致、启动无 mapper 错误，并重测跨租户、deleted、部门/本人权限。

八、灰度发布与回滚预案

1. 灰度发布步骤

  1. 预发环境完成静态门禁、功能回归、SQL 抽样和 Arthas 验证。
  2. 生产先发布单实例或小流量实例，固定路由或按实例版本区分观测。
  3. 观察 30 分钟：确认无 mapper 错误、无 SQL 语义异常、CPU 无新增突刺。
  4. 观察 60 分钟：覆盖常规业务流量，确认 parse total / avg-rt 与接口 P95/P99 达标。
  5. 观察 120 分钟或业务高峰：确认高峰下 JSqlParser 热点没有回升。
  6. 逐步扩大流量至全量。

2. 回滚触发条件

  • 出现跨租户数据返回。
  • 出现 deleted=true 角色关系返回。
  • 出现部门/本人数据范围越权。
  • 出现 BindingException、Invalid bound statement、mapper XML 加载错误。
  • 数据库慢 SQL 或扫描行异常放大。
  • CPU 未改善且出现新的性能瓶颈。

3. 回滚方式

硬性要求

  • 只能整体回退上一版本制品，或关闭该精确 mapper 改造路径。
  • 不能只回滚 Java class 或只回滚 XML。
  • 不能通过添加 @TenantIgnore、tenantLine = "true"、ignore-tables 或全局关闭数据权限来止血。
  • 回滚后需重启/替换实例，以清理本地 JSqlParser parse cache 与类加载状态。

4. 回滚后复核

  • 确认所有实例版本一致。
  • 确认启动无 mapper 绑定错误。
  • 复测跨租户隔离、逻辑删除、部门/本人数据范围。
  • 复核 CPU 是否回到旧热点；如回到旧热点，配合扩容、限流或临时摘除异常实例止血。

九、第二阶段优化方向

1. 基于 mappedStatementId 继续治理

第一阶段上线后，通过 Arthas 继续收集剩余进入 DataPermissionInterceptor.beforeQuery 的高频 mappedStatementId。 后续只能选择满足以下条件的 SQL 加入白名单：

  • 高频。
  • 不承载部门/本人数据范围。
  • 不涉及跨租户聚合。
  • 能明确保留租户隔离和逻辑删除语义。
  • 可以通过 SQL/结果集回归验证。

2. 业务缓存

可评估缓存 tenantId + sorted(roleIds) -> userIds/UserRoleVO，但不建议一开始缓存最终员工列表。 如果缓存最终用户结果，key 必须包含登录人数据范围上下文，否则会有越权风险。

3. JSqlParser 缓存调优

如果减少无效 parse 后，旧实例仍在 JsqlParseCache.getStatement() 有明显热点，再评估 JsqlParserGlobal.setJsqlParseCache(...)、缓存容量、TTL 或缓存实现。该动作应排在 mapper 短路和业务缓存之后， 因为盲目调大缓存可能增加堆压力、GC 和反序列化成本。

十、仍需确认的问题

  1. 生产实际加载的 DataPermissionConfiguration 是否就是当前源码中规则为空的版本？是否有 starter 或其他分支注册了额外规则？
  2. AdminUserServiceImpl.getUserListByRoleCodes 的业务语义是否明确为“当前租户内角色成员全量可见”，还是也需要当前登录人部门/本人数据范围？
  3. AdminUserApiImpl.getCrmUserRoleIdListByRoleIds、BPM/异步/Dubbo 链路是否始终具备正确 TenantContextHolder？
  4. roleService.selectByCodes 返回角色是否已受当前租户约束？role code / role id 在多租户间是否可能复用？
  5. 生产 system_user_role 是否有支撑 tenant_id + role_id + deleted 或等价条件的索引？
  6. 高峰流量下 roleIds 集合大小分布如何？是否存在超大 IN 列表导致 SQL parse cache key 基数过高？

十一、最终建议

建议按“小步、单点、可验证、可回滚”原则推进。第一阶段只改 UserRoleMapper.getUserListByRoleIds，把 default helper 改为明确 mapped statement，并只添加 @InterceptorIgnore(dataPermission = "true")。该改造能优先降低报告中最明确热点链路的数据权限解析成本， 同时把租户隔离、逻辑删除、部门/本人数据范围风险控制在可验证范围内。

若第一阶段验证有效，再依据线上 mappedStatementId 热点清单扩展治理范围。不要在没有运行时证据前扩大 ignore 范围， 也不要优先通过调大 JSqlParser 缓存解决问题。
